# 要件定義書: おでかけ記録×育成アプリ(仮称)

- バージョン: 1.4(ドラフト)
- 作成日: 2026-07-16 / 更新日: 2026-07-18
- ステータス: 壁打ち結果の統合版 + 詳細ディスカバリー + 3観点レビュー(ゲームデザイン/エンジニアリング/戦略)の反映版。詳細設計の起点とする

---

## 1. プロジェクト概要

### 1.1 コンセプト
行った場所を「その場で撮った写真+タグ+メモ」で記録すると、可愛いキャラクターが仲間になり育っていくアプリ。
**記録が主・ゲームは従**。ゲーム要素は「記録を続けたくなる後押し」であり、記録体験を邪魔しない。

### 1.2 解決したい体験
- 行った場所の思い出が Google マップやカメラロールに散らばって振り返れない
- 記録アプリは三日坊主になりがち(続ける動機がない)
- カップル・旅行グループで「共通の思い出」を一緒に貯める場所がない

### 1.3 ターゲットユーザー
- プライマリ: 20〜30代のカップル・友人グループ(旅行・おでかけ好き)
- セカンダリ: 個人でおでかけ記録を残したい人

### 1.4 類似アプリとの差別化
| アプリ | 参考にする点 | 本アプリとの違い |
|---|---|---|
| Google マップ(タイムライン) | 訪問履歴の地図表示 | 受動的な履歴。本アプリは能動的な記録+育成 |
| ポケモンGO / Pikmin Bloom | 図鑑・レア遭遇・育成 | 位置ゲーが主目的。本アプリは記録が主 |
| BeReal | 「その場」の希少性 | 日常共有。本アプリは場所の思い出に特化 |
| Setlog | 共同で1つのログを作る体験、自動ダイジェスト | Vlog特化。本アプリは地図×キャラ |

## 2. 設計原則(全機能がこれに従う)

1. **記録ファースト**: チェックイン(撮影→登録)は3タップ以内で完了する。ゲーム演出はその後に付いてくる
2. **義務を作らない**: デイリーストリーク・毎日ログインボーナスは実装しない。通知は週1回まで(義務感による燃え尽き・離脱を防ぐ)
3. **過剰報酬の回避**: 外発的報酬を盛りすぎて内発的動機(思い出を残したい気持ち)を侵食しない。数値インフレ禁止
4. **被り・再訪も無駄にしない**: どんな行動にも小さな意味がある(おやつ変換・行きつけ認定)
5. **1人でも完結、共有でより楽しい**: ソーシャル機能がなくても成立する設計
6. **確率課金(ガチャ)は導入しない**: ガチャ型の確率課金は心理学的に依存性の高い変動比率強化(variable-ratio reinforcement)であり、原則2・3と正面から矛盾する。レア入手率の偏りは課金ではなくゲーム内メカニクス(10章の遠出ボーナス等)で緩和する

## 3. 機能要件

### 3.1 MVP(v1)

#### F-01 認証
- メール+パスワード / Google / Apple サインイン(Supabase Auth)

#### F-02 地図・ホーム画面
- Google Maps 上に自分の訪問済みスポットをピン表示(MVPでは1ユーザー1個の個人スペースが自動生成される。招待による複数人共有はv2 [3.3])
- 相棒キャラが地図画面に表示される(愛着形成)
- ピンをタップで記録詳細(写真・タグ・メモ・日付・獲得キャラ)

#### F-03 チェックイン(コア機能)
- 現在地から半径100m程度の POI 候補を Places API で取得し、リストから選択(POIスナップ方式)
- POI候補が取得できない場合(API障害・オフライン等)は手動スポット名入力でチェックインを許可し、GPS座標は保持したまま復帰後にPOIを解決する(4章)
- **その場で撮影した写真のみ**登録可能(カメラ起動のみ、ライブラリ選択不可)
- タグ(自動: ジャンル / 手動: 自由タグ)とひとことメモを任意入力
- 制約: 同一スポット(Google Place ID基準)は1日1回まで(2回目以降は記録のみ有効、XP・キャラ抽選の対象外)、1日のチェックイン上限10件
- 「1日」はチェックイン時の端末ローカルタイムゾーンの暦日で判定する(UTC時刻と端末ローカル日付を両方記録)。同一スポットへの連続チェックインは物理的に同一タイムゾーン内で発生するため、時差による判定の穴は生じない
- 上限10件到達時はチェックインボタンを無効化し「今日の記録上限に達しました」を表示(記録自体を失わせないため、翌日以降の再訪を案内)

#### F-04 アルバム登録(過去の思い出)
- 過去の写真+場所を手動登録できる。地図には表示されるが**育成対象外**(ゲームバランス保護)

#### F-05 ジャンル判定
- Google Places の place type をアプリ内13系統にマッピング(マスタテーブルで管理、コード変更なしで調整可能)
- 系統: カフェ・スイーツ / グルメ / 日常 / ショッピング / 自然・公園 / 神社仏閣・歴史 / ミュージアム・アート / 遊び・エンタメ / スポーツ / 動物・いきもの / 温泉・サウナ / 絶景・展望 / 旅の玄関
- 複数 type が付く場合はレア度の高い系統を優先

#### F-06 レア度(系統と分離、スポット単位)
- レア度は place type ごとに設定(★1〜★4)。例: カラオケ★1、遊園地★2、theme_park(ディズニー/USJ級)★4
- 主要スポットは個別マスタで上書き可能

#### F-07 キャラ遭遇・入手
- チェックイン(XP・抽選対象のもの)では**毎回必ず結果が出る**: 新規キャラ入手、または被り→おやつ変換のいずれか。「何も出ないハズレ」は存在しない(原則4)
- レア度は10章の出現率テーブル(合計100%)で抽選し、系統×レア度に対応するキャラが出現する
- 抽選はサーバー側で実行する(4章)。オフラインチェックインは同期時に抽選され、「記録は保存済み、キャラは電波復帰後のお楽しみ」という体験になる
- 被り: 重複入手せず「おやつ」(育成素材、ランク制。10章)に変換

#### F-08 育成・進化
- チェックインで獲得したXPは、**同系統の全所持キャラに全額加算**される(割り振り操作は不要。原則1の3タップ制約を守る)
- XP+おやつ消費で3段階進化(具体的な閾値・計算式は10章)
- 相棒設定: 1匹を相棒にして地図画面に表示(現在地マーカー=相棒)
- 初期相棒: スターター配布はせず、**最初に入手したキャラが自動で相棒になる**。初チェックインまでは通常の現在地マーカーを表示(CHARACTERS.md 3章)

#### F-09 図鑑
- 13系統×レア度のコレクション一覧。未入手はシルエット表示(コンプ欲の可視化)
- マスタ設計原則: **全系統×全レア度にキャラを必ず配置し「空マス」を作らない**(抽選結果に対応キャラが存在しない状態を排除する)
- 初期規模: 13系統×4レア度=52種×3進化段階=**156アセット**

#### F-10 自分基準ボーナス(コールドスタート対応)
- 遠出: ホームエリアから20km/100km/300kmで XP×1.5/×2/×3+レア率UP(具体的な倍率は10章)
- 初めての市区町村: XP+50%、遭遇確定
- 初めての都道府県: ★3以上確定+記念バッジ
- 初系統: 図鑑に新系統解放
- 行きつけ: 同一スポット5回/10回で称号・着せ替えアイテム
- **スタック規則**: 複数のXPボーナスが同時成立した場合(例: 300km遠出+初市区町村)は**最も高い倍率1つのみ適用**し、乗算・加算スタックはさせない(原則3の数値インフレ禁止)。確定系ボーナスは「初都道府県(★3以上確定)>初市区町村(遭遇確定)」の優先順で1つだけ適用
- ホームエリアの変更は90日に1回まで、変更前に確認ダイアログを表示(ボーナスのインフレ濫用防止。原則3・7章)

#### F-11 アカウント削除
- 設定画面(マイページ)からセルフサービスでアカウント削除が可能
- 削除対象: DB上の関連データ(チェックイン・所持キャラ・バッジ・イベントログ等)に加え、**Supabase Storage上の写真オブジェクトも削除**する(DBレコード削除だけでは写真が残るため明示)

### 3.2 画面構成・タブ

4タブ構成: **地図(ホーム) / 図鑑 / アルバム / マイページ**。詳細な画面一覧・遷移・空状態/エラー状態は11章を参照。

### 3.3 v2 以降(実装しないが設計時に考慮する)
- スペース共有: 招待リンクで参加。スポット・アルバム共有+「共有キャラ」を全員で育成
- 旅ログ自動生成: 連続チェックインをルート+写真+獲得キャラのカード/短尺動画に自動編集
- シェア画像生成: 旅ログ・バッジをキャラ入り縦長画像で書き出し(Instagram ストーリー対応、招待QR入り)→ 獲得チャネル
- コミュニティデイ: 月1「◯◯系統の日: 出現率3倍」
- 捕獲ミニゲーム、天気・時間帯別出現テーブル、季節限定キャラ
- グローバル穴場判定(全ユーザーの登録数を使ったレア度。ユーザー数が増えてから)
- 閲覧専用Web版: 地図・図鑑・アルバムの閲覧のみをブラウザで提供(Instagramと同様のモバイルファースト構成。チェックイン・撮影はネイティブアプリ専用のまま)
- **v2共通の設計前提(プライバシー)**: スペース共有・シェア画像・旅ログなど「自分以外に見える」機能は、ホームエリアを他者に一切見せないこと、および自宅周辺のチェックインを共有対象から除外できる「プライバシーゾーン」の実装を前提条件とする。プライバシーゾーンなしで共有機能をリリースしない

### 3.4 将来検討事項(事業化アイデア・詳細未確定)
- 人流データ・観光業向けデータ活用、画像解析による研究応用: チェックインの構造化位置データ(緯度経度・系統・日時)を匿名化した上での分析基盤化。個人情報保護法上の同意取得・匿名化パイプラインなど、MVPとは別次元の要件が必要になるため、現時点ではv1・v2いずれのスコープにも含めない。着手する場合は本書とは別に事業要件を起こす

## 4. 非機能要件

| 項目 | 要件 |
|---|---|
| プラットフォーム | iOS / Android(Expo による単一コードベース)。**Webはv1対象外**(react-native-mapsがWeb非対応のため。将来は閲覧専用Web版を検討 [3.3]) |
| 配布範囲(v1) | 身内限定(TestFlight内部テスト / Google Play 内部テストトラックのみ。いずれも審査不要で配布可能)。一般公開は対象外 |
| 抽選・XP計算 | **サーバー側(PostgreSQL関数 / Edge Function)で実行**し、クライアントは結果を受け取るのみ(改ざん耐性・制約判定の一元化)。オンライン時は登録と同時に抽選し即時演出、オフラインチェックインは同期時に抽選 |
| オフライン | チェックインはオフライン時ローカルキューに保存→復帰時に順次自動送信。送信失敗時は再試行しユーザーに通知。キャラ抽選は同期時にサーバー側で実行されるため、同期時に「同一スポット1日1回」制約に抵触した場合も矛盾なく処理できる(先に記録された1件のみXP・抽選対象、以降は記録のみ有効) |
| POI取得不可時 | Places API障害・電波不良時も手動スポット名入力でチェックインを許可(GPS座標は保持)。POI情報は復帰後に解決し、系統・レア度が確定した時点でサーバー側抽選を実行する。**解決は自動優先**: 復帰後にGPS座標+スポット名でPlaces APIを自動照会し、候補が一意に定まれば自動解決。候補が複数・0件の場合のみユーザーがスポット詳細から候補を選んで解決する(原則1: ユーザーへの追加操作要求は例外時のみ) |
| 位置情報・カメラ権限 | 利用は「使用中のみ」権限。バックグラウンド追跡はしない(プライバシー原則)。権限を拒否した場合はチェックイン機能全体を無効化し、設定画面への導線を表示 |
| 写真 | Supabase Storage。アップロード時にリサイズ(長辺2048px目安)で容量・通信量を抑制。EXIF位置情報はアップロード時に削除(位置情報はcheckinsテーブルの構造化データで別途管理するため写真側に持たせる必要がない) |
| 応答性能 | チェックインフロー(POI取得→撮影→登録)を通常回線で10秒以内 |
| コスト | 個人開発前提。Supabase 無料枠+Google Maps 無料クレジット内で試験運用 |
| セキュリティ | Supabase RLS(Row Level Security)で自分/所属スペースのデータのみアクセス可。マスタテーブル(characters_master・genre_mapping等)は全ユーザー読み取り専用、書き込みは管理者のみ |
| 不正対策 | 位置偽装の完全対策はMVPでは行わない(将来: 移動速度チェック等)。ホームエリア変更は90日に1回までに制限し、遠出ボーナスの濫用を防止。**POIメタデータ(place type・座標・住所)もv1ではクライアント申告値を受け入れる**(身内配布のための割り切り。理論上place type偽装で高レアを詐取可能)。一般公開時はEdge Function経由のサーバー側Place Details照会に移行する(12章の前提条件) |
| コンテンツモデレーション | タグ・メモの自由入力に簡易NGワードフィルタを適用。**フィルタはサーバー側(process_checkin・update_checkin_note RPC内)で適用する**(クライアント側のみではRPC直接呼び出しで迂回可能なため) |
| 個人情報・自宅の保護 | ホームエリア座標・チェックイン位置はRLSで本人(所属スペース)のみ参照可。写真はEXIF除去+非公開バケット。**v2の共有・シェア機能は「ホームエリアの非共有」と「プライバシーゾーン(自宅周辺の記録を共有・書き出しから除外)」を設計前提とする**(Stravaの自宅特定問題の前例) |
| 計測 | Supabase内に自前のeventsテーブルを設け、主要イベント(チェックイン・進化・図鑑開放等)を記録。外部分析基盤(AWS/GCP等)の導入は必要になった時点で検討 |

## 5. データ概要(詳細はDB設計書へ)

主要エンティティ:
- users(ユーザー、ホームエリア座標・ホームエリア最終変更日時を保持。90日クールダウン判定に使用)
- spaces(スペース: MVPでは1ユーザー1個の個人スペースを自動生成。複数人参加はv2)/ space_members
- spots(訪問スポット: Google Place ID をユニークキーとして同一性判定。座標、系統、レア度)
- checkins(チェックイン: spot、写真、タグ、メモ、獲得XP、UTC日時+端末ローカル日付・タイムゾーン、オフライン同期ステータス、抽選ステータス[未抽選/抽選済み])
- albums(過去写真の登録)
- characters_master(キャラ図鑑マスタ: 系統、レア度、進化段階、画像)
- user_characters(所持キャラ: レベル、経験値、相棒フラグ)
- genre_mapping(place type→系統・レア度・系統基礎XPマスタ)
- badges / user_badges(称号・実績)
- snacks / user_snacks(おやつ: ランク[★1〜4対応]、経験値換算値)
- events(計測用イベントログ: user_id、event_type、payload、created_at)

位置検索(近傍POI・訪問済み判定)は PostGIS を使用。

## 6. 技術スタック(確定)

| レイヤ | 技術 | 備考 |
|---|---|---|
| フロント | Expo(React Native)+ TypeScript | iOS/Android 単一コード(Webはv1対象外。4章) |
| 地図 | react-native-maps(Google Maps SDK) | 品質重視で Google 採用。Web非対応のためWeb版を作る場合は別実装が必要 |

| BaaS | Supabase(PostgreSQL + PostGIS / Auth / Storage / RLS) | 抽選・XP計算はPostgreSQL関数/Edge Functionでサーバー側実行(4章)。それ以外のバックエンドコードは最小化 |
| キャラ画像 | 基準1枚を確定→AI生成で展開 | 初期規模52種(13系統×4レア度)×3進化段階=計156アセット。後から差し替え前提 |
| 将来のAPI層 | FastAPI(必要になった時点で追加) | 旅ログ自動生成など重い処理用 |

## 7. リスクと対応方針

| リスク | 対応 |
|---|---|
| Places API のコスト超過 | POI取得結果をキャッシュ、無料クレジット監視、超過時は取得半径・件数を絞る |
| ゲーム要素の過剰化による記録体験の毀損 | 設計原則2・3を機能追加時のレビュー基準にする |
| キャラ絵の制作コスト | AI生成+仮絵で開発を止めない。差し替え可能な設計 |
| 初期ユーザーの位置情報への不安 | 権限は使用中のみ、共有範囲はスペース内のみを明示 |
| 個人開発のスコープ膨張 | v1機能リスト外は原則やらない。追加は本書の改訂を経る |
| ガチャ・課金要素の要望が今後出た場合 | 原則6に基づき導入しない。レア入手の偏りは10章のボーナス設計で緩和する |
| ホームエリアの頻繁な変更によるボーナス濫用 | 変更を90日に1回に制限し、変更前に確認ダイアログを表示 |
| 共有データからの自宅・行動パターン特定(v2) | プライバシーゾーンを実装するまで共有・シェア機能を出さない(3.3節の設計前提)。ホームエリアは常に非共有 |

## 8. 成功指標(仮)

- 自分たち(開発者+パートナー/友人)が1ヶ月使い続けられる ← 最初の指標はこれで十分
- チェックイン完了までの操作が3タップ以内
- 週1回以上のチェックインが4週続くユーザー比率(将来)
- 計測は自前のeventsテーブルの記録をもとに算出する(4章)

**v1完成の定義**: 以下をすべて満たす状態
1. F-01〜F-11が動作し、テストを網羅している
2. TestFlight内部テスト / Google Play内部テストトラック(いずれも審査不要)で身内数名に配布し、試験運用を開始できる
3. 将来の一般公開を見据え、ストア審査基準に準拠した実装を満たしている: プライバシーポリシーURLの用意(簡易版)、位置情報・カメラ権限の利用目的説明文言、アカウント削除導線(F-11)、簡易利用規約(正式な法務レビューは一般公開時に実施 [12章])

## 9. 用語集

| 用語 | 定義 |
|---|---|
| チェックイン | 現在地+その場撮影による記録。育成対象 |
| アルバム登録 | 過去写真の記録。育成対象外 |
| スペース | 記録とキャラを共有する単位。MVPでは1ユーザー1個の個人スペースが自動生成される。招待による複数人共有はv2 |
| 系統 | キャラのジャンル分類(13種)。図鑑の棚 |
| レア度 | スポット・キャラの希少度(★1〜4)。系統とは独立 |
| おやつ | 被りキャラの変換先。育成・進化素材。入手元キャラのレア度に応じて★1〜4のランクがあり、経験値換算値(1/3/8/20)が異なる |
| 系統基礎XP | genre_mappingで管理する系統ごとのチェックイン基礎XP。初期値は全系統10で統一し、後からコード変更なしで調整可能 |
| 行きつけ | 同一スポットへの再訪実績による認定 |
| ホームエリア | 遠出ボーナスの基準となる自宅周辺エリア。変更は90日に1回まで |

## 10. ゲームバランス数値(v1確定値)

### 10.1 XP計算式
チェックインで得られるXP = 系統基礎XP(genre_mappingで管理、初期値は全系統10) × レア度倍率(★1=1 / ★2=2 / ★3=3 / ★4=4) × ボーナス倍率(遠出・初訪問等、10.3節)

### 10.2 進化条件

| 段階 | 累積XP | おやつ換算値合計 |
|---|---|---|
| 2段階目 | 100以上 | 10以上 |
| 3段階目 | 300以上 | 35以上 |

おやつ換算値: ★1由来=1 / ★2由来=3 / ★3由来=8 / ★4由来=20(ランクが混在していても合計値で判定できる)

体感目安(★1スポット中心・ボーナスなしの場合): 2段階目は約10〜15チェックイン、3段階目は約30チェックインで到達する想定。図鑑コンプ後は全チェックインが被り=おやつ化するため(F-07)、おやつ供給は後半に自然と増える。

### 10.3 出現率

基本出現率(ボーナス適用前)と遠出ボーナス適用後のシフト:

| レア度 | 基本 | 20km遠出 | 100km遠出 | 300km遠出 |
|---|---|---|---|---|
| ★1 | 70% | 65% | 50% | 30% |
| ★2 | 22% | 25% | 30% | 30% |
| ★3 | 7% | 8% | 15% | 25% |
| ★4 | 1% | 2% | 5% | 15% |

テーブルは合計100%であり、毎チェックイン(XP・抽選対象のもの)で必ずいずれかのレア度が抽選される(F-07。「何も出ない」結果は存在しない)。全系統×全レア度にキャラを配置し空マスを作らないため(F-09)、抽選結果に対応キャラが不在という状態は発生しない。

初めての都道府県は既存仕様通り★3以上確定(F-10)。ガチャ・課金による確率購入は原則6により導入しない。抽選処理はサーバー側で実行する(4章)。

## 11. 画面一覧・遷移(詳細は SCREEN.md へ)

### 11.1 オンボーディング(初回起動時)
1. サインアップ/ログイン(メール・Google・Apple)
2. 位置情報・カメラ・通知の権限リクエスト(各権限の利用目的説明画面をOSダイアログの前に挟む)
3. ホームエリア設定(現在地から自動設定、この場で調整も可能)
4. 地図画面へ

### 11.2 タブ構成と主要フロー

| タブ | 画面遷移 |
|---|---|
| 地図(ホーム) | 地図(ピン表示。**現在地マーカー自体が相棒キャラ**)→ スポット詳細。チェックインは地図右下のFABから起動: **タップ1=FAB → タップ2=POI選択 → タップ3=シャッター(=登録完了)**。抽選演出はフルスクリーン(ワンタップでスキップ可)→ 結果画面でタグ/メモを任意追記(原則1の3タップを厳守) |
| 図鑑 | 図鑑一覧(系統×レア度) → キャラ詳細(おやつを与える・進化[ユーザーの明示操作で発動]・相棒設定) |
| アルバム | アルバム一覧 → アルバム登録フロー(写真選択[ライブラリ可] → 場所入力 → タグ/メモ) |
| マイページ | プロフィール/ホームエリア変更(90日クールダウン表示) → 称号・バッジ一覧 → 通知設定 → アカウント削除 → 利用規約 |

### 11.3 空状態・エラー状態

| 状況 | 挙動 |
|---|---|
| 図鑑未入手キャラ | シルエット表示(F-09) |
| アルバム0件 | 「過去の思い出を登録しよう」導線を表示 |
| チェックイン前(初回) | 地図に「最初のチェックインをしてみよう」導線 |
| POI候補0件/API障害 | 手動スポット名入力へフォールバック(4章) |
| 位置情報/カメラ権限拒否 | チェックイン機能を無効化し、設定導線を表示(4章) |
| オフライン中 | チェックインはローカルキュー保存、同期待ちインジケーターを表示 |

## 12. 法務・運用(v1範囲)

- 配布範囲: 身内限定(TestFlight内部テスト / Google Play 内部テストトラック。いずれも審査不要)。ただしv1完成定義(8章)として、将来の一般公開審査に通る状態(簡易プライバシーポリシー・簡易利用規約・権限説明文言)を整えておく。正式な法務レビュー・規約整備は一般公開時に実施

- 写真のEXIF位置情報: アップロード時に削除(4章)
- コンテンツモデレーション: タグ・メモに簡易NGワードフィルタを適用
- 計測: 自前eventsテーブルに主要イベントを記録(4章・8章)
