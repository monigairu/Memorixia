## 3. テスト方針(v1「テスト網羅」の定義)

| レイヤ | 方針 |
|---|---|
| RPC関数(process_checkin・feed_snacks・evolve_character・update_home_area等) | **結合テストで網羅**(pgTAPまたはローカルSupabaseに対するテストスクリプト)。特に抽選の確率系は乱数を固定できる設計にし、境界(上限10件・1日1回・90日クールダウン・進化閾値)を全ケース検証 |
| コアフロー(チェックイン) | E2Eテスト1本(Maestro等)+実機実地テスト |
| その他のUI | 手動テスト表(SCREEN.md 8章のエラーマトリクスをチェックリスト化) |
| マスタ整合性 | 空マス検証・rarity_weights合計100検証をシード投入時に自動実行 |
| セキュリティ | RLSの結合テスト(他ユーザーのデータが読み書きできないこと)/ service roleキーがクライアントバンドルに含まれないこと / Storage署名URLの有効期限設定の確認。M6の完了条件に含める |

## 4. リスクと先出しの検証(スパイク)

| リスク | 対応(いつ) |
|---|---|
| react-native-maps+Expoの相性問題 | M0で実機表示まで確認してから他に着手 |
| Places APIの無料枠・料金体系 | M0で最新のSKU別無料枠を確認し課金アラート設定 |
| カメラ→アップロードの実機挙動(EXIF・リサイズ) | M3(b)で最初に単体検証 |
| AI生成の品質・一貫性 | M2並行のスタイルアンカーで早期に見極める。破綻する場合は外部イラスト依頼への切替をM4末までに判断 |
| 週10時間を確保できない | M5を切り離して後送り(コア体験への影響最小)。M3は分割済みなので中断・再開に耐える |
